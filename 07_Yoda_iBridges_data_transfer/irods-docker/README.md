## Docker connection to Yoda using iRODs commands

LC Oct 2025



## Readme first!

Below there are the steps to initiate a connection to a Yoda server from an Ubuntu 22.04 (jammy) docker container. This means that when you close the container, everything you did - including the donwloaded data - is lost.

The final aim - not implemented here - would be to be able to run the container to use icommands and save them to your drive. I left some notes below in the last paragraph about how this could be done.



## TL.DR

Yoda is very picky about the version of irods to install. As far as we know, only version 4.3.3 is working with the current Yoda installation of irods server. However, the latest version (5.0.2) is not retro-compatible, and the binaries for the previous version appear not be available.

After repeated failures it turned out that it is necessary to install also the `irods-runtime` of the same version, and in this case the connection will work. 

Also important is the `"irods_authentication_scheme"` in the `~/.irods/irods_environment.json`, which _must_ be assigned value `"pam"` (_not_ pam_password)

Finally, make sure the port `1247` is open for communication. Otherwise ask your ict to open it



## Installing irods 4.3.3 on an Ubuntu 22.04 container

**NB**: I call the container `irods` for simplicity. If you want to stop the container and restart the procedure at some point, you need to delete it first using `docker rm irods`

```bash
docker run -it --name irods ubuntu:22.04

# Install some binaries we will need later
apt update
apt install -y wget gnupg lsb-release apt-transport-https nano openssl libssl-dev ca-certificates iputils-ping


# Add the source where irods binaries can be found
mkdir -p /etc/apt/keyrings

wget -qO - https://packages.irods.org/irods-signing-key.asc | \
    gpg \
        --no-options \
        --no-default-keyring \
        --no-auto-check-trustdb \
        --homedir /dev/null \
        --no-keyring \
        --import-options import-export \
        --output /etc/apt/keyrings/renci-irods-archive-keyring.pgp \
        --import

echo "deb [signed-by=/etc/apt/keyrings/renci-irods-archive-keyring.pgp arch=amd64] https://packages.irods.org/apt/ $(lsb_release -sc) main" | tee /etc/apt/sources.list.d/renci-irods.list

apt-get update

# List versions available for install
apt list -a irods-icommands

# CHOOSE THE VERSION 4.3.3 AND MAKE SURE YOU INSTALL ALSO THE RUNTIME**
apt-get install irods-icommands=4.3.3-0~jammy irods-runtime=4.3.3-0~jammy

```





## Prepare the configuration file

This should have a specific name and in a specific directory inside your home: `~/.irods/irods_environment.json`



Replace the following values below with your own:

- `irods_home`

- `irods_host`

- `irods_user_name`

- `irods_zone_name`

  

```bash
# ---------- prepare ~/.irods/irods_environment.json ----------------

cd
mkdir .irods
cd .irods
nano irods_environment.json

# NB: The irods_authentication_scheme MUST be "pam", _not_ "pam_password"
{
    "irods_authentication_scheme": "pam",
    "irods_client_server_negotiation": "request_server_negotiation",
    "irods_encryption_algorithm": "AES-256-CBC",
    "irods_encryption_key_size": 32,
    "irods_encryption_num_hash_rounds": 16,
    "irods_encryption_salt_size": 8,
    "irods_home": "/leiden/home",
    "irods_host": "leiden-yoda.irods.surfsara.nl",
    "irods_port": 1247,
    "irods_user_name": "f.ayres.ribeiro@fsw.leidenuniv.nl",
    "irods_zone_name": "leiden"
}
```



## Connect

Once all this is done you can initiate a session with

```bash
iinit
```

You will be asked your pw. After that you can start using icommands, e.g.

`ils`

## Making it usable as a docker container to save data on your drive

To finalize this tool, one way to proceed would be the following:

- create a bash script with all the instructions to install icommands
- copy it inside the container and launch it at runtime
- create a local `.irods/irods_environment.json` and map it to the `root/.irods/irods_environment.json` directory inside the container (remember that you launch the container you are root)
- create a local `data` folder and map it to `/data` inside the container
- remember to also map the local port to the remote port `1247`
- run the container with `--it /bin/bash` so that you can enter into it and use icommands from inside the container

An attempt to do this is sketched in the files in this folder, however it's largely untested.
